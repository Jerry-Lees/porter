# Porter — Feature Ideas

**Porter** is a dual-pane terminal file manager for homelabs and sysadmins. Named after the
person who carries your luggage between places and packs it up for you — porter does the
same for server config files. It works over SSH/SFTP between any two hosts, and can pack
selected files into a `.tar.gz` deployment archive instead of (or in addition to) a live
destination filesystem.

Porter stands on its own as a general-purpose tool, but is designed to integrate with
**labinator** — specifically for building the deployment archives that labinator extracts
onto newly provisioned LXC containers and VMs as part of app-profile deployments.

> Spiritual successor to Midnight Commander (`mc`). If `mc` is the classic, porter is the
> homelab-native evolution: SSH-native, archive-aware, and built for the way sysadmins
> actually work today.

## Dual-Pane SSH/SFTP File Browser

The core of porter: two side-by-side panes, each showing a filesystem. Navigate, select,
and transfer files between any combination of local and remote hosts.

### Pane types

| Pane type | Description |
|---|---|
| Local filesystem | The machine porter is running on |
| Remote host (SSH/SFTP) | Any host reachable via SSH key auth |
| tar.gz archive | Browse or build a `.tar.gz` file (see below) |

### Layout

```
┌─────────────────────────────┬─────────────────────────────┐
│ [L] root@proxmox02:/etc     │ [R] myserver.tar.gz         │
├─────────────────────────────┼─────────────────────────────┤
│ ..                          │ ..                          │
│ nginx/                      │ etc/                        │
│   nginx.conf      644 root  │   nginx/                    │
│   sites-enabled/            │     nginx.conf              │
│   sites-available/          │     sites-enabled/          │
│ ssh/                        │ opt/                        │
│   sshd_config     600 root  │   myapp/                    │
│ cron.d/                     │     config.yaml             │
├─────────────────────────────┼─────────────────────────────┤
│ F3 View  F4 Edit  F5 Copy  F6 Move  F7 Mkdir  F8 Del  F10 Quit │
└─────────────────────────────────────────────────────────────────┘
```

### Key bindings (mc-style)

| Key | Action |
|---|---|
| Tab | Switch active pane |
| Enter | Open directory / view file |
| F3 | View file contents |
| F5 | Copy selected file(s) to other pane |
| F6 | Move selected file(s) to other pane |
| F8 | Delete selected file(s) |
| F10 | Quit |
| Space | Select/deselect file |
| Ins | Select/deselect file and move down |
| Ctrl+A | Select all |
| Ctrl+R | Refresh pane |

### Implementation notes

- Built with **Textual** (Textualize) for the TUI — modern, actively maintained, Python-native.
- SSH/SFTP via **paramiko** (already a labinator dependency — reuse it).
- Each pane is independently navigable; the active pane is highlighted.
- Connection panel on startup: enter hostname (or pick from known hosts via `~/.ssh/config`
  and `~/.ssh/known_hosts`), SSH key selection, optional jump host.
- Transfers show a progress bar (rich `Progress`) with speed and ETA.
- Permissions and ownership are displayed in the file listing and preserved on transfer.

## tar.gz Archive Pane

Instead of a live destination filesystem, the right (or left) pane can be a `.tar.gz`
archive being built or browsed. This is porter's killer feature and the primary integration
point with labinator.

### Modes

**Build mode** — start with an empty archive, add files from the other pane. The archive
grows as you copy files into it. Save to disk when done.

**Browse mode** — open an existing `.tar.gz` and navigate its contents as if it were a
live filesystem. Copy files out to the other pane for inspection or editing.

**Edit mode** — open an existing archive, add or remove files, save back.

### Permissions and ownership in archives

`tar` preserves permissions (chmod bits) and numeric uid/gid by default. Porter does the
same — what you see in the listing is what lands in the archive and what gets restored on
extraction.

**The uid/gid caveat:** numeric ids must mean the same thing on the destination host. In
practice, packages create system users with consistent ids across distros, so if labinator
installs the required packages before extracting the archive, ownership is correct. For
custom users and service accounts created manually, porter writes a **sidecar manifest**
(`labinator-manifest.yaml`) alongside the archive listing any non-standard users/groups,
so labinator can create them before extraction.

### Archive manifest (sidecar file)

```yaml
# labinator-manifest.yaml — generated by porter alongside the .tar.gz
archive: pihole.tar.gz
created: 2026-03-08T14:22:00
source_host: pihole-reference.lees-family.io
custom_users:
  - name: pihole
    uid: 999
    gid: 999
    shell: /usr/sbin/nologin
    home: /var/lib/pihole
custom_groups:
  - name: pihole
    gid: 999
pre_extract_commands:
  - "useradd -r -u 999 -g 999 pihole"
post_extract_commands:
  - "chown -R pihole:pihole /etc/pihole"
  - "systemctl enable pihole-FTL"
```

Labinator reads this manifest before extracting and handles user creation and post-extract
fixups automatically.

### Implementation notes

- Uses Python's stdlib `tarfile` module — no extra dependencies.
- Archive pane shows file listing with full paths, permissions, owner (name if resolvable,
  uid/gid as fallback), and size.
- Files can be dragged from the live pane into the archive pane (F5 copy).
- Directories are added recursively with structure preserved.
- Archive is written atomically (temp file + rename) to prevent corruption on interrupted saves.

## Connection Manager

A built-in SSH connection manager so you don't have to type hostnames every time.

### Features

- Reads `~/.ssh/config` and `~/.ssh/known_hosts` to pre-populate known hosts.
- Displays hosts in a searchable picker on startup.
- Remembers recently used connections (stored in `~/.config/porter/connections.yaml`).
- Supports jump hosts (`ProxyJump`) from SSH config.
- Each pane has an independent connection — you can have two different remote hosts open
  simultaneously.

## Labinator Integration

Porter is the tool for building the `.tar.gz` deployment archives that labinator uses
for app-profile deployments. The workflow:

1. Stand up a reference container (manually, or via a community helper script on a
   throwaway LXC) and configure the application until it works.
2. Open porter with the reference container on the left and a new archive on the right.
3. Browse the reference container's filesystem, select the config files that make the
   app work (`/etc/pihole/`, `/opt/pihole/`, `/var/lib/pihole/`, etc.).
4. Copy them into the archive — porter preserves paths, permissions, and ownership.
5. Porter generates the sidecar `labinator-manifest.yaml` for any custom users/groups.
6. Save the archive to `~/projects/HomeLab/labinator/app-templates/pihole.tar.gz`.
7. Every future Pi-hole deployment via labinator: install packages → extract archive →
   reboot → running.

### App template storage

Archives intended for labinator live in `labinator/app-templates/`:

```
labinator/
└── app-templates/
    ├── pihole.tar.gz
    ├── pihole-manifest.yaml
    ├── nginx-proxy-manager.tar.gz
    ├── nginx-proxy-manager-manifest.yaml
    └── vaultwarden.tar.gz
```

Porter knows this path and can save directly there when run from the labinator project
directory.
